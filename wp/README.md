# WordPress Application Form and Submissions Plugin

This WordPress implementation contains two pieces:

- `application-form` theme: renders the responsive form with vanilla CSS.
- `application-form-submissions` plugin: owns storage, validation, REST API, `admin-ajax.php` handling, and the admin submissions page.

## Requirements

- WordPress 6.x
- PHP 7.4 or newer
- MySQL or MariaDB
- Local by Flywheel, MAMP, Docker, or another local WordPress environment
- Google Chrome if you want to regenerate Lighthouse reports

## Fresh Machine Setup Overview

1. Create a new WordPress site in your local WordPress tool.
2. Confirm WordPress loads before installing this project.
3. Copy the provided `application-form` theme folder into `wp-content/themes/`.
4. Copy the provided `application-form-submissions` plugin folder into `wp-content/plugins/`.
5. Activate the theme and plugin in WordPress Admin.
6. Visit your local site URL and submit the form.

The examples below use `http://test-wp.local`, but your URL may be different.
Use the URL generated by your own local WordPress environment.

## Database Setup

This project does not require one fixed database host. Use the database
credentials from the WordPress environment on your own computer.

If you use Local, open the site in the Local app and check the **Database** tab.
Use the database name, username, password, host, port, or socket shown there.

Common Local values are:

```txt
Database: local
Username: root
Password: root
Host: localhost
```

If `localhost` does not work in Local, use the socket path or host/port from
your own Local Database tab. Do not copy another machine's socket path.

Examples:

```php
define( 'DB_HOST', 'localhost' );
```

or, if Local gives you a TCP port:

```php
define( 'DB_HOST', '127.0.0.1:PORT_FROM_YOUR_LOCAL_APP' );
```

or, if Local requires a socket on your machine:

```php
define( 'DB_HOST', 'localhost:/your/local/mysql/mysqld.sock' );
```

The submissions table is created by the plugin on activation. With the default
WordPress prefix it is named:

```txt
wp_test_submissions
```

## Setup

Copy the theme to:

```txt
wp-content/themes/application-form
```

The theme folder must include `style.css`, `functions.php`, `header.php`,
`footer.php`, `front-page.php`, `index.php`, and the `assets/` directory.

Copy the plugin to:

```txt
wp-content/plugins/application-form-submissions
```

The plugin folder must include `application-form-submissions.php`.

In WordPress Admin:

1. Go to **Appearance > Themes**.
2. Activate **Application Form**.
3. Go to **Plugins**.
4. Activate **Application Form Submissions**.

On activation, the plugin creates:

```txt
wp_test_submissions
```

If your database prefix is not `wp_`, the table name will be:

```txt
{prefix}test_submissions
```

## Verification

1. Open your local site URL, for example `http://test-wp.local`.
2. Submit the empty form and confirm validation errors appear.
3. Fill first name, last name, email, optional fields, and check terms.
4. Submit the form.
5. Confirm the success message appears.
6. Open WordPress Admin.
7. Click **Form Submissions** in the sidebar.
8. Confirm the new row appears.
9. In Adminer or another database client, open `wp_test_submissions` and confirm the row exists.

The WordPress frontend submits through classic WordPress AJAX:

```txt
wp-admin/admin-ajax.php
```

The plugin also registers the REST endpoint required by the backend task:

```txt
POST /wp-json/application-form/v1/submissions
```

## Architecture

```txt
wp-content/
  themes/
    application-form/
      front-page.php            Form markup
      functions.php             Theme setup and asset enqueueing
      header.php                Minimal WordPress header
      footer.php                Minimal WordPress footer
      index.php                 Fallback template
      assets/
        css/form.css            Vanilla CSS recreation of the design
        js/form.js              Client-side validation and AJAX submit
        images/                 Form illustration and icons
  plugins/
    application-form-submissions/
      application-form-submissions.php
```

Theme responsibilities:

- Render the responsive frontend.
- Enqueue the Exo font, CSS, and JS.
- Pass `admin-ajax.php`, action name, and nonce to the frontend.
- Keep presentation separate from storage.

Plugin responsibilities:

- Create the `test_submissions` custom table on activation.
- Validate server-side data.
- Save submissions to the custom table.
- Register the REST endpoint.
- Register `admin-ajax.php` handlers for logged-in and logged-out visitors.
- Render the **Form Submissions** admin page.

## Extending the Form

To add a field end to end:

1. Add the input markup in `themes/application-form/front-page.php`.
2. Add styling if needed in `themes/application-form/assets/css/form.css`.
3. Add the field to `fields` and `getPayload()` in `themes/application-form/assets/js/form.js`.
4. Add client-side validation in `validate()` if the field is required.
5. Add a column to the table schema in `afs_activate()`.
6. For existing installs, add a plugin upgrade routine or run a migration manually.
7. Read and sanitize the posted value in `afs_handle_ajax_submission()`.
8. Read and sanitize the REST value in `afs_create_submission()`.
9. Add the value to `afs_save_submission()`.
10. Add the column to the admin table in `afs_render_admin_page()` if admins should see it.

When adding server-side validation, keep it in `afs_validate_submission()` so the AJAX and REST paths stay consistent.

When changing storage, keep database writes in `afs_save_submission()` so both submission paths use the same insert logic.

## Lighthouse

Reports are saved in:

```txt
wp-content/lighthouse-reports/wordpress.json
wp-content/lighthouse-reports/wordpress.html
```

Latest generated scores:

```txt
Performance: 96
Accessibility: 96
Best Practices: 76
SEO: 92
```

Regenerate the JSON report:

```bash
npm exec --yes --package lighthouse -- lighthouse http://test-wp.local --chrome-path="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --chrome-flags="--headless --no-sandbox --disable-gpu" --output=json --output-path="./wp-content/lighthouse-reports/wordpress.json" --quiet
```

Regenerate the HTML report:

```bash
npm exec --yes --package lighthouse -- lighthouse http://test-wp.local --chrome-path="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --chrome-flags="--headless --no-sandbox --disable-gpu" --output=html --output-path="./wp-content/lighthouse-reports/wordpress.html" --quiet
```

Replace `http://test-wp.local` with your own local WordPress URL. On Windows or Linux, replace `--chrome-path` with the Chrome executable path for that machine, or remove it if Lighthouse can find Chrome automatically.
